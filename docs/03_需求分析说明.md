# 需求分析说明

## 1. 问题定义

现有工具两极分化：系统日历应用功能复杂、学习成本高；备忘录类应用缺少时间维度和进度追踪。个人用户需要一个既轻量又具备时间结构的计划管理工具，能在手机和桌面上流畅使用。

## 2. 目标与价值

| 目标 | 价值 | 衡量方式 |
|---|---|---|
| 支持年 / 月 / 日三级视图 | 满足不同粒度的计划查看需求 | 三个视图均可正常访问和操作 |
| 快速创建计划（< 30 秒） | 降低记录门槛，提升使用频率 | 表单字段精简，日期可预填 |
| 移动端可用 | 随时随地查看和管理 | 768px 以下布局正确切换 |
| 完成状态追踪 | 看到执行进度，形成正向反馈 | 年视图月卡片展示完成率 |
| 支持循环计划 | 固定周期任务无需重复创建 | 每天 / 每周 / 每月批量生成，各实例独立 |
| 优先级可视化 | 一眼识别重要计划 | PlanCard 左侧色条区分三档优先级 |
| 关键词搜索 | 快速找到跨日期计划 | 输入关键词返回匹配结果 |
| 时间轴拖拽 | 直觉式调整日程时间 | 日视图拖拽松手后时间自动保存 |

## 3. 用户场景

| 场景 | 用户 | 当前痛点 | 期望结果 |
|---|---|---|---|
| 早上记录今日计划 | 个人用户 | 打开系统日历操作繁琐 | FAB 一键新建，标题填完即保存 |
| 月末回顾完成情况 | 个人用户 | 无法快速统计本月完成率 | 年视图月卡片直接看 done/total |
| 安排某天时间块 | 个人用户 | 备忘录无时间轴，无法直观排布 | 日视图时间轴展示，拖拽调整时间 |
| 手机上查看今天计划 | 个人用户 | 桌面端日历在手机上挤压变形 | 移动端自动切换为列表模式 |
| 设置每周固定任务 | 个人用户 | 每周都要手动重复创建 | 循环计划一次设置，批量生成 |
| 找到某个历史计划 | 个人用户 | 不记得具体日期，翻日历很慢 | 搜索框输入关键词直接找到 |

## 4. 范围定义

### 本次实现

- 计划的增删改查（标题、日期、时间段、标签、完成状态、优先级）
- 年 / 月 / 日三级视图，视图间导航联动
- 标签系统：5 个预置标签 + 用户自定义（最多 20 字符）
- 移动端响应式布局（768px 断点）
- 完成状态双态切换（未完成 / 已完成）
- 时间段校验（结束时间须晚于开始时间）
- 删除二次确认（Popconfirm）
- 空状态、加载骨架屏、错误重试
- **优先级标识**：低 / 中 / 高三档，PlanCard 左侧色条可视化
- **循环计划**：每天 / 每周（指定星期）/ 每月，提前批量生成；编辑 / 删除支持仅此条 / 此条及之后 / 全部三种范围
- **关键词搜索**：LIKE 匹配标题，结果页按日期分组展示
- **日视图拖拽调整时间**：垂直拖拽时间块，5 分钟吸附，松手后自动保存
- **请求错误处理**：请求拦截器统一弹错误提示，操作按钮有 loading 状态

### 本次不实现

- 用户认证 / 多账户
- 计划提醒 / 推送通知
- 标签过滤（标签仅作标注展示）
- 数据导出 / 备份
- 跨天时间段
- 拖拽跨天移动计划日期
- 子任务
- 计划模板

## 5. 业务规则与约束

| 规则 | 说明 | 触发时机 | 处理方式 |
|---|---|---|---|
| 标题必填，最多 100 字符 | 防止空计划和超长标题 | 表单提交时 | 前端 Form 校验，后端 zod 校验 |
| 日期必填，格式 YYYY-MM-DD | 三级视图均依赖日期字段查询 | 表单提交时 | 前端 DatePicker 保证格式 |
| end_time 须晚于 start_time | 避免无意义时间段 | 表单提交时 | 前端提交前校验，返回字段级错误 |
| 时间段不跨天 | 简化时间轴渲染逻辑 | 设计约束 | 时间选择器限制为 HH:MM |
| 预置标签不可删除 | 保证基础分类始终可用 | 删除标签时 | 后端检查 is_preset，返回 TAG_PRESET_READONLY |
| 自定义标签名唯一 | 避免重复标签混淆 | 创建标签时 | 数据库 UNIQUE 约束，返回 TAG_ALREADY_EXISTS |
| 自定义标签最多 20 字符 | 防止标签过长影响 UI | 表单输入时 | 前端 maxLength + 后端 zod max(20) |
| 过去日期可以选择 | 支持补录历史计划 | 设计决策 | DatePicker 不限制日期范围 |
| done 字段存储为 0/1 整数 | SQLite 无原生 boolean | 数据库层 | Repository 层转换，Service 层透明 |
| tags 字段存储为 JSON 字符串 | SQLite 无数组类型 | 数据库层 | Repository 层序列化 / 反序列化 |
| priority 默认值为 1（低） | 新建计划默认低优先级 | 创建时 | 后端 DEFAULT 1，前端 initialValue={1} |
| 循环计划提前批量生成 | 不做动态展开 | 创建时 | 同一事务内 INSERT 所有实例，共享 recurrence_group_id |
| 循环计划批量修改不改日期 | 日期是各实例的唯一标识 | 批量编辑时 | 前端 date 字段在批量模式下禁用 |
| 循环计划生成上限 | 避免无限生成 | 创建时 | recurrence_end_date 空时默认 90 天，最多 365 条 |

## 6. 开放问题与人工确认

| 问题 | 影响 | 当前决策 | 后续处理 |
|---|---|---|---|
| 月视图移动端如何展示 | 日历在小屏可读性差 | 切换为日期分组列表 | 可后续增加周视图过渡 |
| 年视图是否支持新建计划 | 年视图定位为统计概览 | 仅展示统计，FAB 保留但新建不预填月份 | 可在月卡片内增加快捷入口 |
| 标签删除后已关联计划如何处理 | 数据一致性 | 标签从计划 tags 数组中软移除（暂不实现自动清理） | 后续可加级联清理逻辑 |
| 循环计划各实例完成状态是否共享 | 影响使用体验 | 各实例独立（推荐） | 已实现 |
| 循环计划生成方式 | 动态展开 vs 批量预生成 | 提前批量生成（推荐） | 已实现 |

## 8. 风险与依赖

| 风险 / 依赖 | 影响 | 应对措施 |
|---|---|---|
| better-sqlite3 需要 C++ 构建工具 | Windows 环境安装失败 | 提前安装 windows-build-tools；package.json 声明 onlyBuiltDependencies |
| nvm-windows Node 版本切换 | node-gyp 用错版本编译原生模块 | 明确要求 Node 20，安装命令文档化 |
| SQLite 无并发写保护 | 多进程同时写可能损坏数据 | WAL 模式缓解；单用户场景风险极低 |
| 循环计划批量 INSERT 数量 | 大量插入导致性能问题 | 限制上限 365 条；SQLite 事务批量写入速度可接受 |
| @dnd-kit 与 React 版本兼容 | 拖拽功能不可用 | @dnd-kit/core 6.x 支持 React 18，已验证 |
| 前端无单元测试 | 重构时回归风险 | 当前阶段依赖手工验收和 E2E；后续可补充 vitest |
